# npm Link Plugin Workflow

## Summary

Support using this repo as an OpenCode plugin through `npm link` instead of symlinking individual source files into `~/.config/opencode/plugins`.

## Key Implementation Changes

- Add package entrypoint metadata so `oc-bash-guard` resolves to `bash-guard.ts`.
- Keep the existing `server` export in `bash-guard.ts`; it already matches OpenCode package plugin expectations.
- Replace README local install instructions with `npm link` commands.
- Document using `npm link --save oc-bash-guard` from `~/.config/opencode`.
- Document adding `"plugin": ["./node_modules/oc-bash-guard"]` to OpenCode config.
- Document removing old plugin file symlinks to avoid duplicate loading.

## Tests Or Verification

- Run `bun test`.
- Verify node_modules path import resolves with `bun -e 'const m = await import("./node_modules/oc-bash-guard/bash-guard.ts"); console.log(typeof m.server)'` from `~/.config/opencode` after linking.
- Manually verify in OpenCode after install with safe, prompted, and blocked commands as already documented.

## Decisions Made

- Use TS source entrypoint directly for local development.
- Do not add a build step or `dist/` output yet.
- Use npm link, not bun link, because OpenCode installed through npm does not use Bun's link registry.
- Use the actual linked `node_modules` path in OpenCode config.
- Keep package metadata; no plugin implementation changes needed.

## Tradeoffs And Risks

- TS source entrypoint is simplest for local `npm link`, but less publish-ready than a built `dist/` package.
- Linked package depends on OpenCode being able to load TS package entries from `node_modules`.
- `npm link --save` is needed so future `npm install` runs in `~/.config/opencode` do not prune the linked dependency.
- Old symlinked plugin files can cause duplicate plugin loading if not removed.
